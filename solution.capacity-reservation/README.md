# Introduction
Capacity Reservation module can deploy these resources:
* azurerm_capacity_reservation (required)

> **_NOTE:_** This module is designed for a single Capacity Reservation Group per deployment. It takes a list of VM configurations (the same format used by `mit.virtual-machine`), filters VMs with `use_capacity_reservation = true`, and automatically creates the necessary capacity reservations within the specified group. If you need multiple reservation groups, use separate module calls - one per group.

Example variables structure is located in [variables.md](variables.md).

Example use case is located in [test-case/locals.tf](test-case/locals.tf).

Terraform documentation:

# Terraform Import
There are a few things you need to do to import resources into .tfstate. In the example below there are resources which can be imported within the module. You may need to modify these commands to the OS on which they will be running (Refer to the [documentation](https://developer.hashicorp.com/terraform/cli/commands/import#example-import-into-resource-configured-with-for_each) for additional details).

### Capacity Reservation
* terraform import '`<path-to-module>`.azurerm_capacity_reservation.capacity_reservation["`<capacity-reservation-name>`"]' '/subscriptions/`<subscription-id>`/resourceGroups/`<resource-group-name>`/providers/Microsoft.Compute/capacityReservationGroups/`<capacity-reservation-group-name>`/capacityReservations/`<capacity-reservation-name>`'

 > **_NOTE:_** `<path-to-module>` is terraform logical path from root. e.g. _module.capacity\_reservation_

 > **_NOTE:_** `<capacity-reservation-name>` is automatically generated by the module in the format `cr-{size}-zone{N}` (e.g. `cr-standard-d4s-v3-zone1`). Underscores in VM size are replaced with hyphens and the name is lowercased.

# Outputs
## Structure

| Output Name | Value    | Comment                          |
| ----------- | -------- | -------------------------------- |
| outputs     | name     |                                  |
|             | id       |                                  |
|             | sku_name | VM size for this reservation     |
|             | capacity | Number of reserved instances     |
|             | zone     | Availability zone (can be null)  |

## Example usage of outputs
In the example below, outputted _id_ of the deployed Capacity Reservation module is used as a value for the _scope_ variable in Role Assignment resource.
```
module "capacity_reservation" {
    source = "git@github.com:Seyfor-CSC/mit.capacity-reservation.git?ref=v1.0.0"
    config = [
        {
            capacity_reservation_group_id = module.crg.outputs["crg-prod"].id
            location                      = "northeurope"
            vm_config = [
                {
                    name                     = "SEY-TERRAFORM-NE-VM01"
                    size                     = "Standard_D4s_v3"
                    zone                     = "1"
                    use_capacity_reservation = true
                }
            ]
        }
    ]
}

data "azurerm_client_config" "azurerm_client_config" {
}

resource "azurerm_role_assignment" "role_assignment" {
    scope                = module.capacity_reservation.outputs.cr-standard-d4s-v3-zone1.id # This is how to use output values
    role_definition_name = "Contributor"
    principal_id         = data.azurerm_client_config.azurerm_client_config.object_id
}
```

# Module Features
## Automatic VM Aggregation
This module automatically aggregates VMs by size and zone. Pass the same VM configuration that you use for the Virtual Machine module, and the module will automatically:
1. Filter VMs with `use_capacity_reservation = true`
2. Aggregate VMs by size + zone
3. Calculate capacity (number of VMs of each type)
4. Create Capacity Reservations with correct names (e.g. `cr-standard-d4s-v3-zone1`)

No manual transformation is needed - use the same `locals` for both `mit.virtual-machine` and `mit.capacity-reservation` modules.

## Lifecycle Management
This module uses `create_before_destroy` lifecycle to handle VM size changes gracefully. When a VM size changes, the new capacity reservation is created before the old one is destroyed, ensuring continuous reservation availability.

**Important:** When using this module with the Virtual Machine module, ensure `depends_on = [module.capacity_reservation]` is set on the VM module. This ensures correct operation ordering:
- **Create/Update:** Capacity reservations are created/updated BEFORE VMs
- **Destroy:** VMs are destroyed FIRST, then capacity reservations are updated/destroyed

## Monitoring tags in `ignore_changes` lifecycle block
We reserve the right to include tags dedicated to our product Advanced Monitoring in the `ignore_changes` lifecycle block. This is to prevent the module from deleting those tags. The tags we ignore are: `tags["Platform"]`, `tags["MonitoringTier"]`.

# Known Issues
## `depends_on` is required on the Virtual Machine module
When using this module together with `mit.virtual-machine`, you **must** add `depends_on = [module.capacity_reservation]` to the VM module call. Without it, Terraform may attempt to delete or reduce capacity reservations while VMs are still assigned to them, which will fail due to Azure constraints. The `depends_on` ensures correct ordering: capacity reservations are created before VMs and VMs are destroyed before capacity reservations.

## `config` variable must be a list of objects but only the first item is processed
The `config` variable uses `list(object({...}))` type to maintain consistency with the `mit.virtual-machine` module's input format. However, the module only processes the first item (`var.config[0]`) because it is designed to manage capacity reservations for a single Capacity Reservation Group (for now). If you need to manage capacity reservations across multiple groups, deploy separate module instances — one per group.

## All VMs must be from a single location
This module assumes all VMs in the configuration belong to the same Azure location. The `location` parameter is defined once at the top level of the `config` object and applies to all capacity reservations created by the module. If your deployment includes VMs across multiple locations, you must use separate module calls for each location — each with its own Capacity Reservation Group and corresponding VM subset.
